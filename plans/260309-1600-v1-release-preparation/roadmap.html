Phase 0 — Deep Dive

0. Data Loss Emergency Fix

Prevent SwiftData from silently destroying user data on schema change
P0 Critical Must Ship Swift
This is the single most dangerous bug in the codebase. Any user who updates to a build with the new CustomField models will silently lose all their tasks, AI modes, and settings. There is zero user feedback — data just vanishes on app quit.

Why This Exists

When CustomFieldDefinitionModel and CustomFieldValueModel were added to the SwiftData schema, no VersionedSchema or MigrationPlan was created. On launch, SwiftData detects a schema mismatch with the existing database. The current code catches this error and silently falls back to an in-memory container. The app works normally during the session — but nothing persists. On quit, all data is gone.

What We Have Now (Broken)

  • TaskManagerApp.swift:96-111 — try/catch creates in-memory container on ANY ModelContainer error
  • No VersionedSchema definitions exist anywhere
  • No MigrationPlan for V1→V2 schema change
  • SwiftData uses its default store path (not explicit)
  • No backup before schema migration
  • No error UI — user sees normal app with empty data

What Changes (Before → After)

Before
Schema mismatch → silent in-memory fallback → data loss on quit.
No versioning, no backups, no error UI.
Store at SwiftData default path (opaque).
→
After
StrataSchemaV1 / V2 with lightweight migration.
Explicit storeURL at ~/Library/Application Support/Strata/strata.store.
Auto-backup before migration (max 5 rotating).
DataErrorView shown on failure with retry/reset/contact support.
Post-init integrity check (COUNT query).

Impact if Skipped

Catastrophic. Every existing user who updates will lose all data. No recovery possible (no backup exists). This would be a reputation-ending event for a productivity app. There is no scenario where this can be deferred.

Who is Affected

  • Every existing user who has tasks saved and updates to the new version
  • New installs are fine (no existing database to conflict)
  • This is Swift-only — backend is not involved
TaskWhatFilesWhy This Specific Change
0-1 Create VersionedSchema V1/V2 + MigrationPlan new SchemaVersioning.swift SwiftData needs explicit schema versions to know how to migrate. V1 = original 3 models, V2 = adds 2 CustomField models. Lightweight migration handles the addition automatically.
0-2 Explicit storeURL + pre-migration backup ModelContainer+Config.swift Default SwiftData path is opaque and unreliable for backup/restore. Explicit URL lets us backup the .store + WAL + SHM files before migration. Rotating max 5 backups prevents disk bloat.
0-3 Remove in-memory fallback + DataErrorView TaskManagerApp.swift The root cause — the try/catch that silently creates in-memory container. Replace with error state that shows DataErrorView with retry/reset/contact support actions.
0-4 Wire MigrationPlan into ModelContainer init ModelContainer+Config.swift Connect the schema versioning to the actual container initialization. Without this, the MigrationPlan exists but is never used.
0-5 Data integrity check on launch ModelContainer+Config.swift Safety net: after migration, verify a COUNT query works. If 0 tasks but backup has data, prompt user about potential restore. Catches edge cases the migration might miss.

Phase 1 — Deep Dive

1. Critical Security

Fix 6 vulnerabilities that could leak OTP codes, bypass seat limits, or leave revoked users with paid access
P0 Critical Must Ship TypeScript
Two CRITICAL bugs: (1) Debug OTP codes are returned in API responses — anyone can see the verification code without checking email. (2) No webhook handlers for license revocation or refunds — revoked/refunded users keep paid access forever.

Why These Exist

The auth system was built iteratively during development. Debug shortcuts (returning OTP in response) were added for testing and never removed. The rate limiter was written with an optimistic return true in the catch block to avoid blocking during development. Webhook handling was implemented for activation events but not for revocation/refund events. Session cleanup was deferred.

What We Have Now (Vulnerable)

  • auth.ts:375-377 — response.debugCode = otpCode leaks verification code in API response
  • auth.ts:102 + resolve.ts:55 — rate limiter returns true (allow) on DB error = fail-open
  • /v1/auth/email/verify has no rate limiting — brute-force OTP possible
  • user-entitlements.ts:155-173 — count-then-insert pattern is racy (TOCTOU) — 3 concurrent requests can all get 2 seats
  • No session limit — user can accumulate unlimited active sessions
  • projector.ts has no handler for license_key.revoked or payment.refunded

What Changes (Before → After)

Before
OTP visible in API response.
Rate limiter fails open on DB error.
Verify endpoint unprotected.
Seat allocation racy (TOCTOU).
Unlimited sessions per user.
Revoked/refunded users keep paid access.
→
After
debugCode deleted from code + types + Swift UI.
Rate limiter returns false (deny) on error = fail-closed.
Verify endpoint: 30 req/min per IP.
Atomic INSERT...WHERE COUNT < limit (D1 single-threaded).
Max 10 sessions; oldest auto-revoked on overflow.
license_key.revoked → set inactive. payment.refunded → downgrade.

Impact if Skipped

  • debugCode: Anyone can authenticate without email access. Effectively no authentication.
  • Fail-open: During D1 outage, all rate limits disabled — abuse is unthrottled.
  • TOCTOU: Users can exceed device seat limits by making concurrent requests. Paid tier limits become meaningless.
  • Missing webhooks: Refunded/revoked users retain paid access indefinitely. Direct revenue loss.

Relationship to Other Phases

  • Runs in parallel with Phase 0 (different codebase: TypeScript backend vs Swift app)
  • Phases 2 and 3 depend on this — they modify the same files (auth.ts, projector.ts)
  • Task 1-6 (webhooks) must complete before Phase 2 task 2-2 (tier precedence) can build on it

Phase 2 — Deep Dive

2. High Security

Fix abuse vectors: unprotected restore, tier precedence bugs, webhook races, and unnecessary CORS
P1 High Must Ship TypeScript
These aren't as urgent as Phase 1, but they enable real abuse scenarios. An attacker could spam the restore endpoint (costing Dodo API calls), a race condition in webhooks could produce incorrect entitlement state, and CORS wildcard allows browser-based token theft.

What We Have Now (Exploitable)

  • /v1/restore has no rate limiting — can be hammered (Dodo API cost amplification)
  • Tier precedence is naive: an inactive VIP event overwrites an active Pro entitlement, incorrectly downgrading the user
  • Two webhooks arriving simultaneously for the same user can produce inconsistent state (race condition in projector)
  • Duplicate webhook IDs cause SELECT-then-INSERT to throw 500 errors instead of deduplicating
  • Access-Control-Allow-Origin: * in all responses — unnecessary for a native macOS app

What Changes

  • Restore rate limit: 20/min per IP, 5/min per install_id
  • Tier precedence: State-aware logic — never overwrite active higher-tier with inactive different-tier
  • Webhook race: Application-level guards from tier precedence fix. Atomic SQL deferred unless races observed in production
  • Idempotency: INSERT...ON CONFLICT(webhook_id) DO NOTHING, check meta.changes
  • CORS: corsHeaders() returns {}. Native URLSession doesn't enforce CORS

Impact if Skipped

Restore abuse causes Dodo API cost accumulation. Tier precedence bug causes paying users to lose features when unrelated events fire. Webhook races create support tickets from users seeing incorrect tier. CORS wildcard is an unnecessary attack surface — if session tokens leak via XSS anywhere, browser-based exploitation is possible.

Must or Optional?

Must ship. The tier precedence bug (2-2) directly causes incorrect user state. The restore rate limit prevents cost amplification. CORS removal is trivial and eliminates a class of attacks. All 5 tasks are required before GA.

TaskWhatFilesWhy
2-1 Rate limit restore endpoint routes/restore.ts Unauthenticated restore calls hit Dodo's API. Without limits, attacker can amplify costs. 20/min per IP is generous for normal use (1-2 calls).
2-2 State-aware tier precedence projector.ts:466-472 Current logic: latest event wins. But "VIP cancelled" shouldn't deactivate an active Pro subscription. New logic: (a) never downgrade active higher-tier, (b) inactive higher-tier doesn't overwrite active different-tier, (c) same-tier same-state is no-op.
2-3 Concurrent webhook race fix projector.ts:488-521 Two webhooks processed simultaneously can interleave reads and writes. Start with 2-2's application guards. Atomic SQL CASE approach deferred unless races observed.
2-4 Atomic webhook idempotency routes/webhook.ts:169-190 SELECT-then-INSERT for dedup throws UNIQUE violation on duplicates. Change to INSERT...ON CONFLICT DO NOTHING and check meta.changes. Returns 200 with deduplicated: true.
2-5 Remove CORS headers index.ts:84-91 Strata is macOS-only (URLSession). CORS headers serve no purpose and enable browser-based token exploitation. Replace corsHeaders() body with return {}.

Phase 3 — Deep Dive

3. Quality + Infrastructure

Input validation, operational infrastructure, and key rotation readiness — 12 tasks across Swift + TypeScript
P2 Medium Should Ship Swift + TS
This is the largest phase (12 tasks, 10h). It combines 7 quality/validation fixes with 5 infrastructure improvements. Most tasks are independent and parallelizable. This phase bridges the gap between "secure" and "production-grade."

System Ownership — Who Handles What?

Strata's backend is not a thin wrapper around Dodo. We built a full auth + entitlement stack. Here's the boundary:

Our Backend (CF Workers + D1)
  • User accounts (users table)
  • OTP generation (randomDigits(6))
  • OTP hash + storage (auth_challenges)
  • OTP verification (hash compare)
  • Session tokens (SHA-256 hashed, 30-day)
  • Rate limiting (resolve_rate_limits)
  • Device seat tracking (user_devices)
  • Entitlement tier mapping (product_id → tier)
  • Ed25519 token signing (signing.ts)
  • Webhook processing + idempotency
DodoPayments (3rd Party)
  • Payment processing (credit cards)
  • Checkout sessions
  • Subscription lifecycle (create/renew/cancel)
  • License keys (VIP lifetime)
  • Webhook events (sends to us)
  • Does NOT handle: user accounts, auth, devices, email verification, entitlement logic
Resend (3rd Party)
  • Email delivery only
  • We call their API with rendered HTML
  • Never sees OTP code — only the final email body
  • Does NOT handle: OTP generation, validation, user accounts, rate limiting

Why We Built Our Own Auth (Not Dodo)

Dodo is a payment processor — it handles money, not identity. Dodo knows about customers (email + payment method), but has no concept of:

  • Device tracking — Dodo doesn't know which Mac the user is on. We need install_id for seat limits (Free:1, Pro:2, VIP:3)
  • Session management — Dodo has no bearer tokens. We need per-device sessions for API auth
  • Email verification — Dodo verifies emails at checkout (for payment receipts), but we need to verify email ownership to link a device to an account. This is our OTP flow.
  • Entitlement decisions — Dodo sends product_id strings. We must map them to tiers and decide who gets what features.

Bottom line: Dodo handles the money side. We handle the identity + authorization side. They connect via webhooks.

3-1 Email length validation Our BE
validation.ts		 Our code, our problem. We validate emails in requireEmail() before OTP send. Currently checks format via regex but no length cap. Dodo validates emails separately at checkout time for payment receipts — but that's their flow. Our OTP flow accepts any-length email, which could cause storage issues in our D1 users table.
Fix: Add if (normalized.length > 254) before regex. RFC 5321 max.
3-2 OTP rejection sampling Our BE
auth.ts:141-152		 We generate the OTP ourselves. Our randomDigits(6) function uses crypto.getRandomValues() to get random bytes, then does byte % 10 for each digit. Resend only receives the final 6-digit code inside an HTML email — they never generate it.
Modulo bias: bytes 0-255 mod 10 gives 0-5 each with probability 26/256 (~10.16%) and 6-9 with 25/256 (~9.77%). Trivial but easy to fix.
Fix: Discard bytes ≥ 250, use remainder. Uniform distribution.
3-3 Body size limit (1MB) Our BE
index.ts		 Our middleware, our responsibility. Cloudflare Workers has a 100MB default body limit (their backstop), but we should reject large POSTs earlier. No 3rd party involved.
Fix: Check Content-Length header before route dispatch. >1MB → 413.

index.ts		 Our middleware. All our API endpoints expect JSON, but we don't check Content-Type. An HTML form could POST application/x-www-form-urlencoded and reach our handlers. Webhook endpoint excluded (Dodo sends their own content type).

Impact if Skipped

  • Validation gaps (3-1 to 3-4): Not immediately exploitable but fail security audit. Body size limit prevents resource exhaustion.
  • product_id (3-5): All subscriptions map to "pro" — VIP users get Pro tier. Direct feature-gating bug.
  • No CRON (4-1): DB tables grow unbounded. Performance degrades over months.
  • No key rotation (4-3): Key compromise = invalidate ALL tokens simultaneously, no graceful migration.

Must or Optional?

Task 3-5 (product_id) is effectively P0 — without it, VIP users get Pro tier. The rest are P2 "should ship." Key rotation (4-3) is the most important infrastructure task — incident-response readiness. CRON (4-1) prevents operational debt. Validation tasks are standard hardening.

Data Flow: Dodo Webhook → Our Entitlement System

Dodo sends webhook → POST /v1/webhooks/dodo (our endpoint)
  → We verify HMAC-SHA256 signature (Dodo's signing key)
  → We check idempotency (webhook_id in our webhook_events table)
  → We extract product_id from event payload
  → We lookup PRODUCT_IDS map: "pdt_xxx" → "pro" or "pdt_yyy" → "vip"
  → We upsert user_entitlements table with tier + status
  → We sign an Ed25519 entitlement token (our key, our format)
  → Client app verifies token with our public key + kid

Dodo's job ends at the webhook. Everything after the arrow is ours.

4. Legacy Cleanup + GA Gate

Delete dead code paths, add monitoring, and pass the 19-item GA checklist
P3 Final Must Ship TypeScript
This is the point of no return. After this phase, legacy email-only trust paths are permanently deleted. There's no flag to re-enable them. Every endpoint requires cryptographic authentication. This phase also adds anomaly logging for post-launch monitoring.

Role of This Phase

During development, endpoints had an email-only fallback path gated by feature flags (AUTH_REQUIRED_FOR_RESTORE, etc.). These flags are all true in production, so the legacy paths are unreachable — but the code still exists. It's dormant attack surface: one misconfiguration re-enables email-only trust for paid entitlements. This phase deletes it all.

What We Have Now

  • auth.ts has optionalAuthSession(), authRequiredForRestore(), authRequiredForResolve(), authRequiredForCheckout(), isTruthyFlag() — all flag-related functions
  • types.ts and wrangler.jsonc have flag-related env var definitions
  • No anomaly detection for account sharing signals
  • No monitoring for suspicious patterns (rapid account switching, device bursts)
